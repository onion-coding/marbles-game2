# Provably-fair protocol

How a player verifies after the fact that their round was not rigged.

Status: **implemented** as of M3 (2026-04-15). Code: [game/fairness/seed.gd](../game/fairness/seed.gd) (derivation), [game/main.gd](../game/main.gd) (commit/reveal in the round flow), [game/verify_main.gd](../game/verify_main.gd) (third-party verifier). Replay format v2 ([game/recorder/replay_writer.gd](../game/recorder/replay_writer.gd)) carries `server_seed`, `server_seed_hash`, per-marble `client_seed` + `spawn_slot`, and `slot_count` so a verifier can run with no out-of-band data.

## Goal

For any completed round, anyone (player, auditor, regulator) can independently verify:

1. The server could not have chosen the winner after seeing the buy-ins.
2. Spawn positions were derived deterministically from public inputs and a pre-committed secret.
3. The physics outcome follows from those spawn positions (clients replay the recorded sim and see the same result).

## Inputs

- `server_seed` — 32 random bytes, generated by the server before buy-in opens. **Kept secret until SETTLE.**
- `server_seed_hash` — `SHA-256(server_seed)`. **Published before buy-in closes** (broadcast to all clients in the `BUY_IN` phase start event).
- `round_id` — monotonic integer, public.
- `client_seed` — optional per-player string mixed in at marble spawn. For MVP this is a stub but the hook exists.
- `marble_index` — 0..N for each marble in the round.

## Spawn derivation

For each marble:

```
h = SHA-256(server_seed || round_id (u64 BE) || client_seed || marble_index (u32 BE))
spawn_slot = h[0..4] interpreted as u32 BE, mod SLOT_COUNT
```

`SLOT_COUNT` is a track-specific constant (e.g. 24 slots across the spawn rail). If two marbles hash to the same slot, advance to the next free slot deterministically by scanning `(spawn_slot + 1) mod SLOT_COUNT` until one is free. This resolution rule is part of the protocol — document any change loudly.

## Round lifecycle

1. **WAITING:** server generates `server_seed`, stores it securely, publishes `server_seed_hash` and the track id.
2. **BUY_IN:** players buy in with optional `client_seed`. Marbles are registered but not yet positioned.
3. **RACING (start):** server derives all spawn slots (above), starts the headless sim, records ticks.
4. **SETTLE:** server reveals `server_seed`. Replay is archived with `{server_seed, server_seed_hash, round_id, participants[], tick_data, winner}`.

## Verification (third-party)

Given an archived round, a verifier:

1. Checks `SHA-256(server_seed) == server_seed_hash`. If not, the server tampered — proof of fraud.
2. Re-derives spawn slots from the inputs. Must match the recorded spawn positions.
3. (Optional, harder) Re-runs the sim with those spawns and confirms the recorded tick data is consistent. This requires the same Godot/Jolt build and is **not bit-deterministic across platforms** — see the determinism note in [../PLAN.md](../PLAN.md). Visual/outcome match is the bar, not bit-exactness.

## Threat model — what this protocol does NOT protect against

- Server colluding with a specific player to **choose which `server_seed`** from many pre-generated candidates to commit to. Mitigation: commit to a chain of future seeds in advance (hash chain), or mix in a public beacon (e.g. future Bitcoin block hash) that the server can't predict.
- Server withholding buy-ins after seeing the seed. Mitigation: seed hash published **before** buy-in opens (not just before it closes).
- Client-side cheating. Not applicable — clients are passive replay viewers.

## Open questions

- Hash chain vs single-round commitment? Single-round is simpler; chain gives stronger auditability.
- Do we mix in a public beacon (Bitcoin block, drand)? Adds latency but kills the "choose your seed" attack.
- Storage: how long do we retain replays? Regulatory requirement is usually 6 months to multiple years depending on jurisdiction.
